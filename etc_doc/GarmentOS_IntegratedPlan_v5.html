방법론
PDD + SDD + Harness + ISA-95 + PDC
v4.6의 3축 방법론(PDD·SDD·Harness)에 이번 세션의 ISA-95 기반 PDC 체계를 결합한 5축 통합 방법론입니다.
● PDD
Prototype-Driven Discovery
34개 화면 HTML 프로토타입을 고객에게 직접 시연하여 요구사항을 확정. 오전 피드백 → 오후 Spec 자동 생성. 수동 Spec 5일 → 1~2일로 단축.
✅ 4/13 세션에서 수행 완료
● SDD
Spec-Driven Development
코드 작성 전 requirements.md + design.md + tasks.md 3단계 명세 완성. Claude Code에게 정확한 설계도 제공 → 재작업 최소화. PDC WSD가 이 역할 수행.
✅ PDC WSD 37개 태스크로 대체
● Harness
결함 사전 방어 체계
의류 MES 결함 패턴 38개를 사전 정의. 구현 시 Claude Code가 방어 로직·테스트 자동 포함. SIT 기간 4주→2주 단축 효과.
✅ 38개 카탈로그 v4.6에서 채택
● ISA-95
국제 MES 표준 (Part 3)
Level 4(ERP) · Level 3(MES 핵심) · Level 2(현장 인터페이스) · Level 1(물리 현장)의 4계층으로 시스템을 설계. DB·API 도메인 구조의 기준.
★ 이번 세션 신규 반영
● PDC
Process Data Contract
37개 태스크 × 입력·처리·출력·검증·거버넌스 계약. ERD 테이블 수준으로 정의. CRUD Matrix로 71테이블 커버리지 검증.
★ 이번 세션 신규 반영
◆ 핵심 원칙
"사람은 판단, AI는 실행"
Phase 2b에서 코드의 70~80%를 Claude Code가 생성. 인간은 ① Spec 작성 → ② Gate Review → ③ CR 판단 → ④ 최종 승인에 집중.
🔄 변경 관리 3중 안전망 (v4.6 채택)
① Feature Flag
코드를 지우지 않고 기능을 끈다. 화면 레벨 33개 + 기능 레벨 10개 = 총 43개 Flag 관리. 롤백 시 Flag off만으로 처리.
② Atomic Commit + Worktree
화면당 독립 브랜치. 주간 Git 태그 + DB 스냅샷 동시 생성. 코드 롤백 시 DB도 함께 복원.
③ Spec Versioning
명세서·코드 동일 Git 저장소. 코드 롤백 시 Spec도 자동 복원. PDC WSD가 Spec 역할 수행.
📊 DB 스키마 변경 원칙 (v4.6, CLAUDE.md 필수 기재)
Phase 1 전 기간: Additive-Only 원칙 — DB 컬럼 삭제 금지. 변경 시 기존 컬럼 유지 + 새 컬럼 추가. 불필요 컬럼 정리는 Phase 2 안정화 이후. 이 규칙을 CLAUDE.md에 명시하여 Claude Code도 준수.
